# Repository Polish Checklist

Everything needed to take this repo from "working production app" to "looks like a mature open-source project at a glance." Grouped by effort, with current status where it's checkable from the repo itself. This is a checklist, not an autopilot task — none of these files are auto-generated by this document.

- [x] README with hero, screenshots section (placeholders), live demo links, features, architecture, folder structure, install, env vars, deployment, challenges, future work (`README.md`)
- [x] Architecture doc with diagrams (`docs/portfolio/ARCHITECTURE.md`)
- [x] Case study (`docs/portfolio/CASE_STUDY.md`)

## Missing — recommended before sharing widely

### Legal / community health files
- [ ] **`CODE_OF_CONDUCT.md`** — standard Contributor Covenant is a safe default if this ever takes outside contributions.
- [ ] **`CONTRIBUTING.md`** — even a short one: how to run tests, branch naming, commit convention (a project convention already exists internally in `docs/engineering/04-commit-convention.md` — surface it here for outside contributors).
- [ ] **`SECURITY.md`** — how to report a vulnerability responsibly. Worth having given the project's own attack surface (NL2SQL, public unauthenticated API).

### GitHub-native polish
- [ ] **Badges** in README header — suggested set: build status (once CI exists), license, deployment status (Vercel/Render), language/framework badges (React, FastAPI, TypeScript). Avoid vanity badges (visitor counters, etc.) — they read as noise to engineering reviewers.
- [ ] **GitHub Topics** on the repo settings page — suggested: `fastapi`, `react`, `typescript`, `postgresql`, `pgvector`, `nl2sql`, `text-to-sql`, `semantic-search`, `vector-search`, `llm`, `openrouter`, `rag`, `full-stack`.
- [ ] **Repository description + website URL** in GitHub's own repo metadata — currently likely blank; set to the one-line pitch and the live demo URL.
- [ ] **Social preview image** — GitHub Settings → Social Preview. A 1280×640 image (screenshot collage or a designed card) that shows up when the repo link is shared on LinkedIn/Twitter/Slack. Without one, shared links look bare.
- [ ] **Issue templates** (`.github/ISSUE_TEMPLATE/bug_report.md`, `feature_request.md`) — even minimal ones signal the repo is maintained.
- [ ] **PR template** (`.github/PULL_REQUEST_TEMPLATE.md`) — checklist mirroring the internal `docs/engineering/10-pr-checklist.md` if one exists, so external and future-you contributions follow the same bar.
- [ ] **Releases / tags** — cut a `v1.0.0` tag on the current production commit. Even a single tag gives the repo a versioned reference point and shows up in GitHub's UI as a real release, not just an unversioned `main`.

### Visual assets
- [ ] **Screenshots** — `docs/screenshots/{ask,overview,products,search,visual}.png`, referenced by the README placeholders already in place. Capture at a consistent viewport width (1600px suggested), light mode, with realistic (not empty) data visible.
- [ ] **GIFs** for the two highest-signal interactions: (1) typing a question and watching the SSE stages appear live on the Ask screen, (2) typing a search query and watching hybrid results rank. 10–15 seconds each, under ~5MB (use a tool that outputs optimized GIF/WebP, not a raw screen recording).
- [ ] **Demo video** — a 60–90 second walkthrough covering all 5 screens plus one NL2SQL query end-to-end. Host on YouTube (unlisted is fine) or as an MP4 in a release asset; link it from the README's Live Demo section.
- [ ] **Architecture diagrams as static images** — the Mermaid diagrams in `ARCHITECTURE.md` render natively on GitHub, but exporting the high-level one as a PNG (for the social preview image or a portfolio site that doesn't render Mermaid) is worth doing once the diagram is finalized.
- [ ] **Favicon / app icon** — confirm the deployed frontend has a real favicon (not the default Vite icon) — check `frontend/public/` / `frontend/index.html`.

### Documentation
- [ ] **API reference** — FastAPI's auto-generated `/docs` (Swagger UI) covers this live; consider linking it prominently, or exporting the OpenAPI JSON into the repo (`docs/openapi.json`) so it's browsable without a running server.
- [ ] **CHANGELOG.md** — even a simple "Unreleased / v1.0.0" log going forward, especially once releases/tags exist.
- [ ] See [`DOCUMENTATION_AUDIT.md`](DOCUMENTATION_AUDIT.md) for a full gap analysis beyond this checklist.
